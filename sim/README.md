# G1 Isaac Sim ‚Äî locomotion & localization testbed

Launches **Isaac Sim** with the Unitree **G1** plus a **Livox MID-360** LiDAR
and IMU, published to ROS 2 **exactly the way the real robot does**, so the same
`FAST_LIO_LOCALIZATION_HUMANOID` stack runs unchanged against the simulator.

```mermaid
flowchart TB
    subgraph ISAAC["Isaac Sim ‚Äî Python 3.11 (g1_sim_scene.py)"]
        stage["G1 stage<br/>/World/g1"]
        rtx["RTX LiDAR<br/>Livox_Mid360 config<br/>@ torso_link/mid360_link"]
        imu["IMU sensor<br/>(same mount)"]
        clk["ReadSimulationTime"]
        stage --> rtx
        stage --> imu
    end

    subgraph BRIDGE["g1_sim_bridge ‚Äî ROS 2 Humble / Python 3.10 (SIM ONLY)"]
        adapt["isaac_livox_custom_adapter_node"]
    end

    subgraph STACK["Unchanged navigation stack"]
        flio["FAST-LIO<br/>mid360.yaml ¬∑ lidar_type:1 ¬∑ use_sim_time"]
        rest["A* + MPC + AMO gait"]
    end

    rtx -->|"/livox/lidar<br/>PointCloud2, livox_frame"| adapt
    imu -->|"/livox/imu_raw<br/>Imu (BEST_EFFORT)"| adapt
    clk -->|"/clock"| flio
    adapt -->|"/livox/custom_msg<br/>CustomMsg (RELIABLE)"| flio
    adapt -->|"/livox/imu<br/>Imu (RELIABLE)"| flio
    flio --> rest

    linkStyle 0,1,2 stroke:#888
```

> All Isaac‚ÜíROS 2 traffic is **CycloneDDS** (`RMW_IMPLEMENTATION=rmw_cyclonedds_cpp`),
> the same transport the rest of the stack uses.

The Python-version split (Isaac = 3.11, Humble = 3.10) is why the
PointCloud2‚ÜíCustomMsg conversion is a **separate ROS 2 node**, not done inside
Isaac. The bridge makes Isaac's output indistinguishable from the real Livox
driver, so **`mid360.yaml` is used as-is** ‚Äî no sim/real config fork.

## Why a Livox MID-360 specifically

The config `lidar_configs/Livox_Mid360.json` models the real sensor's geometry:
360¬∞ horizontal √ó **‚àí7¬∞‚Ä¶+52¬∞** vertical FOV, 0.1‚Äì70 m range, ~10 Hz, ~200k pts/s.
It's an RTX **rotary** multi-beam (64 beams spun 360¬∞) ‚Äî *not* the proprietary
non-repetitive rosette, but a same-FOV cloud FAST-LIO consumes identically. The
`rotary` scan type is the well-tested Isaac path that reliably produces returns
(a hand-authored `solidState` rosette emitted empty clouds ‚Äî `width:0`).
Generated by `gen_mid360_config.py` ‚Äî see *Tuning*.

## Run (simulation)

**1 ‚Äî Isaac Sim** (publishes `/livox/lidar`, `/livox/imu_raw`, `/clock`):

```bash
cd Navigation/sim
./launch_g1_sim.sh                 # GUI
./launch_g1_sim.sh --headless      # no window
ROS_DOMAIN_ID=7 ./launch_g1_sim.sh # match your stack's domain
```

**2 ‚Äî bridge + FAST-LIO** (in a Humble-sourced shell with the ws built):

```bash
ros2 launch g1_sim_bridge sim_localization.launch.py
# bridge only (run FAST-LIO yourself):
ros2 launch g1_sim_bridge sim_localization.launch.py start_fastlio:=false
```

Both sides must share `RMW_IMPLEMENTATION=rmw_cyclonedds_cpp` and `ROS_DOMAIN_ID`.

**3 ‚Äî walk.** The G1 in sim is unactuated until you drive its joints. Feed your
locomotion policy / companion the way you do on the robot; FAST-LIO localizes
off the LiDAR+IMU regardless.

## Verifying topics reach the localization container

Isaac runs on the **host**; FAST-LIO runs in the **`localization` container**.
They interoperate over DDS only when **all three** match: `rmw_cyclonedds_cpp`,
the same `ROS_DOMAIN_ID`, and ROS 2 **Humble** (the container is Humble; the
launcher forces Isaac to Humble too ‚Äî a stray `/opt/ros/jazzy` in your shell
otherwise makes Isaac publish on Jazzy typesupport that Humble cannot read). The
container uses `network_mode: host` + default CycloneDDS, so discovery works over
loopback with no extra config.

```bash
# host: start Isaac (Humble + CycloneDDS + domain 0, all forced by the launcher)
Navigation/sim/launch_g1_sim.sh

# container: open a shell and run the bridge + the checker
cd Navigation/docker
docker compose run --rm localization bash
#   inside the container (sim/ is mounted at /sim, the ws at /ws):
source /opt/ros/humble/setup.bash && source /ws/install/setup.bash
ros2 launch g1_sim_bridge sim_localization.launch.py start_fastlio:=false &  # produces /livox/custom_msg
bash /sim/verify_topics.sh
```

[`verify_topics.sh`](verify_topics.sh) lists topics and reports the rate of each
expected one ‚Äî Isaac-raw (`/clock`, `/livox/lidar`, `/livox/imu_raw`) and
bridge-output (`/livox/custom_msg`, `/livox/imu`) ‚Äî and checks the IMU Z. Manual
equivalents:

```bash
ros2 topic list
ros2 topic hz  /livox/lidar          # ~10 Hz
ros2 topic hz  /livox/custom_msg     # ~10 Hz (needs the bridge running)
ros2 topic echo /livox/imu_raw --once --field linear_acceleration   # z ‚âà +9.81 at rest
```

If a topic is `NOT ADVERTISED` from inside the container: check `ROS_DOMAIN_ID`
matches Isaac, that both use `rmw_cyclonedds_cpp`, and that no `CYCLONEDDS_URI`
pins DDS to a NIC that excludes loopback.

## Tuning the MID-360

Density is `N_BEAMS` (vertical) √ó `reportRateBaseHz` (horizontal columns/s),
with `reportRateBaseHz = TARGET_POINTS_PER_SEC / N_BEAMS` (default 64 beams,
200 000 pts/s). After a run, check:

```bash
ros2 topic echo /livox/lidar --field width --once   # points per msg (must be > 0!)
ros2 topic hz /livox/lidar                            # ~10 Hz
```

Edit `N_BEAMS` / `TARGET_POINTS_PER_SEC` at the top of `gen_mid360_config.py`,
re-run it (the launcher also regenerates a missing config and symlinks it into
Isaac's config path), and relaunch.

> If `width:0` (empty clouds): the config didn't load or the scene has no
> RTX-hittable geometry. Isolate with a built-in profile ‚Äî
> `./launch_g1_sim.sh --lidar-config Example_Rotary`; if points appear, it's the
> custom config; if still empty, the stage lacks geometry in range.

## How this matches the real robot

| | Real robot | Simulation |
|---|---|---|
| LiDAR cloud | `livox_ros_driver2` `xfer_format=1` ‚Üí `CustomMsg` | `g1_sim_bridge` ‚Üí `/livox/custom_msg` (`CustomMsg`) |
| IMU | Livox driver ‚Üí `/livox/imu` (RELIABLE) | Isaac ‚Üí `/livox/imu_raw` ‚Üí bridge ‚Üí `/livox/imu` (RELIABLE) |
| Frame | `livox_frame` | `livox_frame` |
| Rate | 10 Hz | 10 Hz |
| FAST-LIO config | `mid360.yaml` (lidar_type:1) | `mid360.yaml` **unchanged**, `use_sim_time:=true` |

On the real robot `g1_sim_bridge` is **not** used ‚Äî the Livox SDK driver emits
`CustomMsg` + `/livox/imu` natively. See the real-deploy notes below.

## Frame & Z-axis conventions (LiDAR + IMU)

FAST-LIO (`mid360.yaml`) assumes `extrinsic_R = identity` and an **upright**
`livox_frame`: cloud Z up, and a right-side-up IMU reading **accel ‚âà (0, 0,
+9.81) m/s¬≤** at rest. The sim is built to land in exactly that state.

```mermaid
flowchart LR
    subgraph REAL["Real robot"]
        rphys["MID-360 mounted<br/>UPSIDE-DOWN"]
        rdrv["livox_ros_driver2<br/>cloud: roll:180 (MID360_config.json)<br/>IMU: R_x(180) (lddc.cpp)"]
        rup["livox_frame UPRIGHT<br/>cloud Z up ¬∑ IMU (0,0,+9.81)"]
        rphys --> rdrv --> rup
    end
    subgraph SIM["Simulation"]
        smnt["mid360_link is UPRIGHT in the URDF<br/>(rpy 0 0.040 0 ‚Äî pitch only, no roll)"]
        ssen["LiDAR + IMU mounted IDENTITY<br/>at mid360_link (co-located)"]
        sup["livox_frame UPRIGHT<br/>cloud Z up ¬∑ IMU (0,0,+9.81)"]
        smnt --> ssen --> sup
    end
    rup -. "same contract" .- sup
```

Why this is correct:

- **The real robot's correction and the sim's mounting converge on the same
  upright frame.** On hardware the sensor is *physically* inverted and the
  driver un-flips both streams (`roll:180` for the cloud, `R_x(180)` for the
  IMU). In sim the sensor is *not* physically inverted ‚Äî `mid360_link` is upright
  in the URDF (`rpy 0 0.0401 0`, a ~2.3¬∞ pitch, **no roll**) and we mount the
  LiDAR + IMU at identity, so the output is already upright. No `roll:180` is
  applied (or needed) in sim.
- **LiDAR‚ÜîIMU stay consistent.** Both are children of the *same* `mid360_link`
  prim with the *same* orientation, so `extrinsic_R = identity` holds exactly as
  `mid360.yaml` assumes ‚Äî and the sim **avoids the real robot's historical
  inverted-IMU bug** (cloud upright + IMU raw-inverted ‚Üí divergence on motion).
- **IMU units & sign match.** Isaac `IsaacReadIMU` runs with `readGravity=True`,
  emitting **m/s¬≤** specific force ‚Äî at rest `(0, 0, +9.81)`, the same convention
  the real `/livox/imu` and the `odometry_debugger` expect (FAST-LIO normalises
  the magnitude regardless).

**Verify on first run** (the one thing worth confirming, since a flipped IMU is
silent until the robot moves):

```bash
ros2 topic echo /livox/imu --once     # linear_acceleration.z should be ‚âà +9.81 at rest
# ground returns should sit BELOW the sensor (z ‚âà ‚àí1 m), ceiling above.
```

If your G1 USD ever mounts `mid360_link` inverted (Z down), reproduce the real
driver's flip with `./launch_g1_sim.sh --lidar-rpy-deg 180,0,0` instead of
editing the stack.

## Real-deploy stack ‚Äî readiness

The Docker deploy (`Navigation/docker`, service `localization`) is unblocked:

- `Dockerfile.localization` fixes (in build order):
  - `FROM osrf/ros:humble-desktop` (the `library/ros` repo has no `desktop` tag).
  - removed `libc++-dev/libc++abi-dev` (they pull `libunwind-14-dev`, which
    `Breaks: libunwind-dev` that `libgoogle-glog-dev` needs).
  - Open3D built with `-DGLIBCXX_USE_CXX11_ABI=ON` so it matches Humble's
    `rclcpp` ABI (else `open3d_loc` fails to link with `Ss`-mangled undefined
    `std::string` symbols).
  - Livox-SDK1 built with `-DCMAKE_CXX_FLAGS="-include memory"` (GCC-11 needs the
    explicit `<memory>` include).
  - After rebuilding the image, **clean the bind-mounted workspace** before
    recompiling (stale old-ABI objects persist on the host):
    `rm -rf /ws/build /ws/install /ws/log && colcon build`.
- Real LiDAR: `livox_ros_driver2/launch/msg_MID360_launch.py` ‚Üí
  `xfer_format=1`, `frame_id=livox_frame`, `publish_freq=10` ‚Äî the CustomMsg +
  IMU that `mid360.yaml` consumes.

Build & run the real localization (per the top-level README):

```bash
cd Navigation/docker
docker compose build localization                 # image
docker compose run --rm localization bash
build_ws                                           # rosdep + colcon inside the container
ros2 launch open3d_loc localization_3d_g1.launch.py
```

> This pass wired the files and ran static checks only; the heavy
> `docker compose build` / `colcon build` were **not** executed here ‚Äî run the
> commands above to produce the binaries.
